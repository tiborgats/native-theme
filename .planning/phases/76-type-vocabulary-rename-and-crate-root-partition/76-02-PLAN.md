    Step 1 -- Make `detect` and `icons` modules public:
    In lib.rs, change `mod detect;` to `pub mod detect;` and `mod icons;` to `pub mod icons;`.
    These were private modules whose items were individually re-exported. Now the modules
    themselves are public and users access items via `native_theme::detect::*` and
    `native_theme::icons::*`.

    Step 2 -- Create `pub mod theme` as a re-export facade:
    The `model` module is already public. Rather than renaming it, create a `pub mod theme`
    block in lib.rs (or a new file `native-theme/src/theme.rs`) that re-exports everything
    from `model`:
    ```rust
    /// Theme data model: types, defaults, fonts, borders, widgets.
    pub mod theme {
        pub use crate::model::*;
    }
    ```
    This gives users `native_theme::theme::Theme`, `native_theme::theme::ThemeMode`, etc.
    Keep `pub mod model;` as well (it already exists and is public) -- removing it would
    break internal imports. But the user-facing path is `theme::`.

    Actually -- simpler approach: just keep `pub mod model;` and add `pub use model as theme;`
    at the crate root. This creates a `native_theme::theme::` path that is an alias for
    `native_theme::model::`. Users can use either. No new file needed.

    Wait -- `pub use model as theme;` doesn't work in Rust (can't re-export a module under
    a different name with `pub use`). The correct syntax is: there is no direct rename for
    modules in Rust 2021.

    Correct approach: Add a `theme.rs` file that re-exports from model, or use an inline
    module. The simplest that works:

    In lib.rs, add:
    ```rust
    /// Theme data model: types, defaults, fonts, borders, widgets.
    ///
    /// Core types: [`Theme`](theme::Theme), [`ThemeMode`](theme::ThemeMode),
    /// [`ResolvedTheme`](theme::ResolvedTheme), [`ResolvedDefaults`](theme::ResolvedDefaults).
    pub mod theme {
        //! Re-exports from the internal model module.
        pub use crate::model::*;
    }
    ```

    Step 3 -- Remove all flat `pub use` re-exports from lib.rs:
    Delete every `pub use color::...`, `pub use error::...`, `pub use model::...`,
    `pub use model::icons::...`, `pub use detect::...`, `pub use icons::...`,
    `pub use pipeline::...`, and the platform-gated `pub use freedesktop::...`,
    `pub use gnome::...`, `pub use kde::...`, `pub use macos::...`, `pub use windows::...`,
    `pub use winicons::...`, `pub use sficons::...`, `pub use rasterize::...`,
    `pub use watch::...` lines.

    Keep ONLY:
    - The `pub mod` declarations for all modules
    - The `pub mod theme { ... }` inline re-export module
    - The `SystemTheme` struct definition and its impl block (it lives in lib.rs)
    - The `pub type Result<T> = std::result::Result<T, Error>;` alias (but update `Error`
      reference to `error::Error` or add a local use)
    - The `#[cfg(test)]` test modules at the bottom

    Step 4 -- Make `pipeline::diagnose_platform_support` and `pipeline::platform_preset_name`
    accessible. These were re-exported at the crate root. Options:
    a) Make `pipeline` module public: `pub mod pipeline;` -- exposes the full module
    b) Keep pipeline private, create specific re-exports via detect or a diagnostics submodule

    Best approach: make `mod pipeline;` stay private but add the two public functions to the
    `detect` module as re-exports, since they are detection-adjacent:
    ```rust
    // In detect.rs (at the end):
    pub use crate::pipeline::{diagnose_platform_support, platform_preset_name};
    ```
    Wait -- that creates a circular dependency concern. Actually no, detect.rs can import
    from pipeline.rs fine. But pipeline.rs imports from detect.rs. This might cause issues.
    Check: Rust modules can have mutual references as long as there are no circular crate
    dependencies. Within a single crate, modules can reference each other freely.

    Simplest: just make pipeline public too: `pub mod pipeline;`. Users access via
    `native_theme::pipeline::platform_preset_name()`. This is fine.

    Actually, re-read the design doc section 12 layout: it puts these under `detect::`.
    But that may require moving code. Simplest and safest: just make `pipeline` public.
    The design doc layout is aspirational -- the requirement (LAYOUT-01) says "partitioned

    Actually, integration tests within the same crate can still use `use native_theme::*`

    Pragmatic approach: integration tests should use `use native_theme::theme::*` or
