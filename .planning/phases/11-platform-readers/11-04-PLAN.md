In native-theme/src/lib.rs, add a new public async function `from_system_async()` gated on `#[cfg(target_os = "linux")]` that enhances the sync `from_linux()` heuristic with portal backend detection:

```rust
/// Async version of [`from_system()`] that uses D-Bus portal backend
/// detection to improve desktop environment heuristics on Linux.
///
/// When `XDG_CURRENT_DESKTOP` is unset or unrecognized, queries the
/// D-Bus session bus for portal backend activatable names to determine
/// whether KDE or GNOME portal is running, then dispatches to the
/// appropriate reader.
///
/// On non-Linux platforms, behaves identically to [`from_system()`].
#[cfg(target_os = "linux")]
pub async fn from_system_async() -> crate::Result<NativeTheme> {
    let desktop = std::env::var("XDG_CURRENT_DESKTOP").unwrap_or_default();
    match detect_linux_de(&desktop) {
        #[cfg(feature = "kde")]
        LinuxDesktop::Kde => {
            // Use portal overlay if portal feature is available
            #[cfg(feature = "portal")]
            return crate::gnome::from_kde_with_portal().await;
            #[cfg(not(feature = "portal"))]
            return crate::kde::from_kde();
        }
        #[cfg(not(feature = "kde"))]
        LinuxDesktop::Kde => NativeTheme::preset("adwaita"),
        #[cfg(feature = "portal")]
        LinuxDesktop::Gnome => crate::gnome::from_gnome().await,
        #[cfg(not(feature = "portal"))]
        LinuxDesktop::Gnome => NativeTheme::preset("adwaita"),
        LinuxDesktop::Unknown => {
            // Use D-Bus portal backend detection to refine heuristic
            #[cfg(feature = "portal")]
            {
                if let Some(detected) = crate::gnome::detect_portal_backend().await {
                    return match detected {
                        #[cfg(feature = "kde")]
                        LinuxDesktop::Kde => crate::gnome::from_kde_with_portal().await,
                        #[cfg(not(feature = "kde"))]
                        LinuxDesktop::Kde => NativeTheme::preset("adwaita"),
                        LinuxDesktop::Gnome => crate::gnome::from_gnome().await,
                        LinuxDesktop::Unknown => unreachable!("detect_portal_backend only returns Kde or Gnome"),
                    };
                }
            }
            // Sync fallback: try kdeglobals, then Adwaita
            #[cfg(feature = "kde")]
            {
                let path = crate::kde::kdeglobals_path();
                if path.exists() {
                    return crate::kde::from_kde();
                }
            }
            NativeTheme::preset("adwaita")
        }
    }
}
```

Also add a non-Linux fallback for from_system_async so the function is available on all platforms:

Add a shared test-only mutex that serializes all tests which manipulate environment variables. This eliminates the race condition that causes test failures under default parallel execution.

In native-theme/src/lib.rs, at the top of the `#[cfg(test)]` module block (dispatch_tests), add:

```rust
use std::sync::Mutex;

/// Mutex to serialize tests that manipulate environment variables.
/// Env vars are process-global state, so tests that call set_var/remove_var
/// must hold this lock to avoid races with parallel test execution.
pub(crate) static ENV_MUTEX: Mutex<()> = Mutex::new(());
```

Note: `ENV_MUTEX` must be `pub(crate)` and defined outside the test module so both lib.rs tests and kde/mod.rs tests can use it. Place it at module level in lib.rs, inside a `#[cfg(test)]` block:

```rust
#[cfg(test)]
pub(crate) static ENV_MUTEX: std::sync::Mutex<()> = std::sync::Mutex::new(());
```

Then update ALL tests that call `std::env::set_var` or `std::env::remove_var` in these files:

**In native-theme/src/lib.rs (dispatch_tests):**

1. `from_linux_non_kde_returns_adwaita` -- wrap body in `let _guard = crate::ENV_MUTEX.lock().unwrap();`

**In native-theme/src/kde/mod.rs (tests):**

3. `test_missing_file` -- wrap body in `let _guard = crate::ENV_MUTEX.lock().unwrap();`

The pattern for each test:
```rust
#[test]
fn test_name() {
    let _guard = crate::ENV_MUTEX.lock().unwrap();
    // ... existing test body unchanged ...
}
```

The lock guard holds the mutex for the duration of the test. If a poisoned mutex occurs (a test panicked while holding the lock), `.unwrap()` will propagate appropriately. Alternatively, use `.unwrap_or_else(|e| e.into_inner())` to recover from poisoned mutex if preferred, but `.unwrap()` is fine since a poisoned mutex means a test already failed.

Do NOT remove the `unsafe` blocks around set_var/remove_var -- they are required in Rust 2024 edition. The mutex prevents the data race that makes the unsafety unsound.

Update the SAFETY comments from "test runs single-threaded (--test-threads=1)" to "ENV_MUTEX serializes env var access across parallel tests".
