**Part A: native-theme-build crate-level docs**

Add a `//!` doc block at the very top of `native-theme-build/src/lib.rs` (before `mod codegen;`). The doc block should follow the pattern used by other crates in the workspace (see native-theme-iced/src/lib.rs for style reference). Include:

1. **Overview** section explaining the crate's purpose: build-time code generation for custom icon roles. Mention it reads TOML definitions and produces a Rust enum implementing `IconProvider`.

2. **TOML Schema** section showing a complete master TOML example:
```toml
name = "app-icon"
roles = ["play-pause", "skip-forward", "volume-up"]
bundled-themes = ["material"]
system-themes = ["sf-symbols", "segoe-fluent", "freedesktop"]
```

3. **Directory Layout** section showing the expected file structure:
```text
icons/
  icons.toml           # Master TOML (the file passed to generate_icons)
  material/
    mapping.toml       # Role -> SVG filename mappings
    play_pause.svg
    skip_next.svg
    volume_up.svg
  sf-symbols/
    mapping.toml       # Role -> SF Symbol name mappings
  segoe-fluent/
    mapping.toml       # Role -> Segoe codepoint mappings
  freedesktop/
    mapping.toml       # Role -> freedesktop icon name mappings
```

4. **Mapping TOML** section showing a simple mapping.toml example:
```toml
play-pause = "play_pause"
skip-forward = "skip_next"
volume-up = "volume_up"
```
And a DE-aware mapping example for freedesktop:
```toml
play-pause = { kde = "media-playback-start", default = "media-play" }
```

5. **build.rs Setup** section showing both APIs:
```rust,ignore
// Simple API (single TOML file):
native_theme_build::generate_icons("icons/icons.toml");

// Builder API (multiple TOML files):
native_theme_build::IconGenerator::new()
    .add("icons/media.toml")
    .add("icons/navigation.toml")
    .enum_name("AppIcon")
    .generate();
```

6. **Using the Generated Code** section showing how to include and use:
```rust,ignore
// In your lib.rs or main.rs:
include!(concat!(env!("OUT_DIR"), "/app_icon.rs"));

// The generated enum implements IconProvider:
use native_theme::{load_custom_icon, IconSet};
let icon_data = load_custom_icon(&AppIcon::PlayPause, IconSet::Material);
```

7. **What Gets Generated** brief section explaining the output: enum + IconProvider impl + include_bytes! for bundled SVGs.

8. **Validation** section noting that build errors are emitted for: missing roles in mapping files, missing SVG files, unknown role names, duplicate roles across files, missing `default` key in DE-aware entries.

Use `//!` comment style throughout. Do NOT use `#[doc = include_str!]` pattern (the build crate doesn't have a README.md and its examples are not runnable as doctests since they need build-time context).

**Part B: IconProvider trait-level docs**

In `native-theme/src/model/icons.rs`, the `IconProvider` trait (around line 343) already has method-level docs and an example. Add a trait-level `///` doc comment ABOVE the existing `/// # Examples` section. The trait-level doc should explain:

1. What the trait represents: the interface for custom icon types that integrate with native-theme's loading system.
2. That implementors map their icon variants to platform-specific icon names and optional bundled SVG data.
3. That `native-theme-build` can auto-generate implementations from TOML definitions.
4. That the trait is object-safe (only requires `Debug` supertrait).
5. That `IconRole` implements this trait, delegating to the built-in icon mappings.
6. Brief mention that `load_custom_icon()` accepts any `impl IconProvider`.

Keep it concise (8-15 lines of doc comment). The existing `# Examples` section already provides code.
