## Summary

Phase 45 implements the resolution engine that converts a sparse `ThemeVariant` (where most fields are `Option<T>` and many are `None`) into a complete `ResolvedTheme` where every field is guaranteed populated. This is the central piece of the v0.5.0 pipeline: OS readers and TOML presets produce partial data, and `resolve()` fills the gaps via ~90 inheritance rules before `validate()` converts to the non-optional output type.

The data model foundation from Phase 44 is complete: 25 per-widget `XxxTheme` structs with `Option<T>` fields, their matching `ResolvedXxx` counterparts (generated by `define_widget_pair!`), `ThemeDefaults`, `FontSpec`, `TextScale`, and `impl_merge!`. What Phase 45 adds is:
1. A `resolve(&mut self)` method on `ThemeVariant` that applies inheritance rules in a defined 4-phase order
2. A `ResolvedTheme` struct (and `ResolvedDefaults`, `ResolvedTextScale`) mirroring `ThemeVariant` but with non-optional fields
3. A `validate()` function that converts `ThemeVariant` -> `Result<ResolvedTheme, ThemeResolutionError>`
4. `ThemeResolutionError` listing every missing field path as a Vec<String>
5. Enrichment of the 10 community preset TOMLs to provide all non-derived fields (fonts, mono_font, line_height, accent_foreground, and additional widget-specific fields)

All inheritance rules are fully specified in `docs/todo_v0.5.1_inheritance-rules.md`. The resolution pipeline architecture is fully specified in `docs/todo_v0.5.1_resolution.md`. No external library research is needed -- this is pure Rust code using only types already in the crate.

**Primary recommendation:** Implement in 3 plans: (1) ResolvedDefaults + ResolvedTextScale + ResolvedTheme struct + ThemeResolutionError, (2) resolve() with all ~90 inheritance rules in 4-phase order + validate(), (3) PRESET-03 preset enrichment + integration tests proving all 17 presets pass resolve() then validate().

### Pattern 1: 4-Phase resolve() Execution Order

**What:** resolve() applies inheritance rules in a strict 4-phase order to avoid dependency cycles and ensure each phase's outputs are available to subsequent phases.

**When to use:** Always -- this is the only correct order.

**Phases:**
```
Phase 1: Defaults internal chains
  - defaults.selection = defaults.selection.or(defaults.accent)
  - defaults.focus_ring_color = defaults.focus_ring_color.or(defaults.accent)
  - defaults.selection_inactive = defaults.selection_inactive.or(defaults.selection)

Phase 2: Defaults safety nets (for fields with no universal derivation)
  - These are last-resort fallbacks that only fire if both OS reader
    and TOML left the field as None. Platform-divergent fields get a
    single reasonable fallback.
  - input.caret = input.caret.or(defaults.foreground)
  - scrollbar.track = scrollbar.track.or(defaults.background)
  - spinner.fill = spinner.fill.or(defaults.foreground)
  - popover.background = popover.background.or(defaults.background)
  - list.background = list.background.or(defaults.background)

Phase 3: Widget-from-defaults (the bulk of ~80 rules)
  - Colors: widget.background = widget.background.or(defaults.X)
  - Geometry: widget.radius = widget.radius.or(defaults.radius)
  - Fonts: resolve_font_spec(&mut widget.font, &defaults.font)
  - FontSpec sub-field inheritance (RESOLVE-04)
  - TextScale entry inheritance (RESOLVE-05)
  - Accent propagation (RESOLVE-06):
    button.primary_bg = button.primary_bg.or(defaults.accent)
    checkbox.checked_bg = checkbox.checked_bg.or(defaults.accent)
    slider.fill = slider.fill.or(defaults.accent)
    progress_bar.fill = progress_bar.fill.or(defaults.accent)
    switch.checked_bg = switch.checked_bg.or(defaults.accent)

Phase 4: Widget-to-widget chains
  - window.inactive_title_bar_background =
      window.inactive_title_bar_background.or(window.title_bar_background)
  - window.inactive_title_bar_foreground =
      window.inactive_title_bar_foreground.or(window.title_bar_foreground)
```

### Pattern 2: FontSpec Sub-Field Inheritance

**What:** Per-widget font fields inherit sub-fields individually from `defaults.font`, not as a whole struct replacement.

**When to use:** For every widget that has an `optional_nested { font: [FontSpec, ResolvedFontSpec] }`.

**Example:**
```rust
/// Resolve a per-widget font from defaults.
/// If the widget font is None, clone defaults entirely.
/// If the widget font is Some but has None sub-fields, fill from defaults.
fn resolve_font(widget_font: &mut Option<FontSpec>, defaults_font: &FontSpec) {
    match widget_font {
        None => {
            // Entire font inherits from defaults
            *widget_font = Some(defaults_font.clone());
        }
        Some(font) => {
            // Individual sub-fields inherit
            if font.family.is_none() {
                font.family = defaults_font.family.clone();
            }
            if font.size.is_none() {
                font.size = defaults_font.size;
            }
            if font.weight.is_none() {
                font.weight = defaults_font.weight;
            }
        }
    }
}
```

Widgets with fonts: window (title_bar_font), button, input, menu, tooltip, toolbar, status_bar, dialog (title_font).

### Pattern 3: TextScaleEntry Inheritance

**What:** TextScaleEntry sub-fields inherit individually from defaults, with line_height computed from a multiplier.

**Example:**
```rust
fn resolve_text_scale_entry(
    entry: &mut Option<TextScaleEntry>,
    defaults_font: &FontSpec,
    defaults_line_height: Option<f32>,
) {
    let entry = entry.get_or_insert_with(TextScaleEntry::default);
    if entry.size.is_none() {
        entry.size = defaults_font.size;
    }
    if entry.weight.is_none() {
        entry.weight = defaults_font.weight;
    }
    if entry.line_height.is_none() {
        if let (Some(lh_mult), Some(size)) = (defaults_line_height, entry.size) {
            entry.line_height = Some(lh_mult * size);
        }
    }
}
```

### Pattern 4: validate() Field-Path Collection

**What:** validate() walks every field of ThemeVariant, collecting field paths of None values into a Vec<String>, then either returns ResolvedTheme or ThemeResolutionError.

**Example:**
```rust
fn validate(variant: &ThemeVariant) -> Result<ResolvedTheme, ThemeResolutionError> {
    let mut missing = Vec::new();

    // Check defaults fields
    let accent = require(&variant.defaults.accent, "defaults.accent", &mut missing);
    let background = require(&variant.defaults.background, "defaults.background", &mut missing);
    // ... ~150 more fields

    if !missing.is_empty() {
        return Err(ThemeResolutionError { missing_fields: missing });
    }

    Ok(ResolvedTheme {
        defaults: ResolvedDefaults {
            accent: accent.unwrap(),
            background: background.unwrap(),
            // ...
        },
        // ...
    })
}

fn require<T: Clone>(
    field: &Option<T>,
    path: &str,
    missing: &mut Vec<String>,
) -> Option<T> {
    match field {
        Some(val) => Some(val.clone()),
        None => {
            missing.push(path.to_string());
            None
        }
    }
}
```

### Pattern 5: ResolvedTheme as Output-Only Type

**What:** ResolvedTheme does NOT implement Serialize or Deserialize. It is output-only -- produced by validate(), consumed by connectors. This prevents accidentally serializing a resolved theme (which would lose the Option semantics needed for merging).

**Derives:** Clone, Debug, PartialEq only. No serde derives.

### Anti-Patterns to Avoid

- **Resolving fields in random order:** Dependencies between inheritance rules mean order matters. Always use the 4-phase structure.
- **Panicking on None in validate():** Must collect ALL missing fields, not fail-fast on the first one. Users need the complete list.
- **Implementing Deserialize on ResolvedTheme:** This is an output type. Input is always ThemeVariant.
- **Hardcoding fallback colors in resolve():** resolve() copies from `defaults.X`, never from hardcoded hex values. Hardcoded values are forbidden per project memory.
- **Using unwrap_or_default() for resolution:** `or(defaults.X)` is the only correct pattern. Default trait values would produce nonsensical themes.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Field-path string generation | Manual string concatenation per field | Consistent `"widget.field"` pattern | ~150 paths need consistency; use a helper function |
| Font resolution for 8 widgets | Copy-paste per widget | `resolve_font()` helper | Identical logic for all 8 font-carrying widgets |
| TextScale entry resolution for 4 entries | Copy-paste per entry | `resolve_text_scale_entry()` helper | Identical logic for caption, section_heading, dialog_title, display |

**Key insight:** The resolution engine is ~90 rules that follow only 4-5 patterns. Factor into helper functions, not 90 individual assignments.

## Common Pitfalls

### Pitfall 1: Dependency Order in resolve()

**What goes wrong:** Setting `button.primary_bg = defaults.accent` before `defaults.selection = defaults.accent` could cause confusion if code were to later depend on selection being set first.
**Why it happens:** The 4-phase structure exists precisely to prevent this.
**How to avoid:** Phase 1 (defaults internal chains) MUST run before Phase 3 (widget-from-defaults). Phase 4 (widget-to-widget) MUST run last.
**Warning signs:** A widget field depending on another widget field that was supposed to be set in the same phase.

### Pitfall 2: FontSpec Resolution Edge Cases

**What goes wrong:** A widget has `font: Some(FontSpec { family: Some("Mono"), size: None, weight: None })`. If resolve() replaces the entire FontSpec with defaults, the explicit family choice is lost.
**Why it happens:** Treating FontSpec as atomic rather than field-level.
**How to avoid:** Always use sub-field inheritance (pattern 2 above). Only replace entire FontSpec when widget.font is None.
**Warning signs:** Test that sets only `menu.font.size = 12.0` loses the default family after resolve().

### Pitfall 3: Missing Fields in validate()

**What goes wrong:** A new field is added to a widget struct but not checked in validate(), causing ResolvedTheme construction to silently skip it or compile-time error.
**Why it happens:** validate() is a manual field-by-field extraction.
**How to avoid:** validate() must mirror every field in every struct. Use a systematic approach: walk ThemeDefaults fields, then each of 25 widgets, then text_scale, then icon_set.
**Warning signs:** `cargo test` passes but a field in ResolvedTheme has a garbage value.

### Pitfall 4: Platform-Divergent Safety Nets

**What goes wrong:** Using `defaults.accent` as the universal safety net for `input.caret` when on KDE it should be `defaults.focus_ring_color` and on GNOME it should be `defaults.accent`.
**Why it happens:** Per-platform differences in caret color derivation.
**How to avoid:** The resolve() safety net uses ONE reasonable fallback (e.g., `defaults.foreground` for `input.caret`). The actual correct value comes from the OS reader or platform TOML before resolve() runs. The safety net is truly last-resort.
**Warning signs:** Cross-platform presets produce visually wrong carets.

### Pitfall 5: Cross-Platform Preset Gaps After Enrichment

**What goes wrong:** A community preset passes resolve() + validate() in tests but looks visually broken because a field was given a nonsensical default.
**Why it happens:** Adding `font.family = "sans-serif"` to a Catppuccin preset that should use "Inter" or similar.
**How to avoid:** Use consistent, reasonable values for community presets. Fonts: "Inter" (14.0) / "JetBrains Mono" (14.0). Line height: 1.4. Weight: 400 (body), 700 (headings). Verify visually if possible.
**Warning signs:** Tests pass but visual inspection shows wrong fonts or sizes.

### Pitfall 6: Integration Test Compilation

**What goes wrong:** The existing integration tests in `native-theme/tests/merge_behavior.rs` reference the old flat structure (`base.colors.background`).
**Why it happens:** Phase 44 restructured the model but the integration tests still use old paths.
**How to avoid:** Phase 45 should update or replace the integration tests that reference old types. The lib unit tests already pass with the new structure.
**Warning signs:** `cargo test -p native-theme --test merge_behavior` fails to compile.

### ThemeResolutionError

```rust
/// Error returned when a ThemeVariant cannot be fully resolved.
#[derive(Debug, Clone)]
pub struct ThemeResolutionError {
    /// Field paths that were still None after resolution.
    pub missing_fields: Vec<String>,
}

impl std::fmt::Display for ThemeResolutionError {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        write!(f, "theme resolution failed: {} missing field(s):", self.missing_fields.len())?;
        for path in &self.missing_fields {
            write!(f, "\n  - {path}")?;
        }
        Ok(())
    }
}

impl std::error::Error for ThemeResolutionError {}
```

## Cross-Platform Preset Gap Analysis (PRESET-03)

### Current State

10 community presets (catppuccin x4, nord, dracula, gruvbox, solarized, tokyo-night, one-dark) and "default" currently provide:
- All ThemeDefaults colors (accent, background, foreground, surface, border, muted, shadow, status colors, selection, link, focus_ring_color, disabled_foreground)
- Geometry (radius, radius_lg, frame_width, disabled_opacity, border_opacity, shadow_enabled)
- Spacing (full scale)
- Some widget colors (button bg/fg, input bg/fg, tooltip bg/fg, sidebar bg/fg, popover bg/fg, separator color, list alternate_row)
- Some widget sizing (button, input, checkbox, tooltip, scrollbar, slider, progress_bar, tab, menu, list, toolbar, splitter)

### Missing for PRESET-03 (needed to pass resolve() + validate())

Fields that community presets must add:
1. **Fonts**: `[defaults.font]` (family, size, weight) and `[defaults.mono_font]` (family, size) -- currently only platform presets + default have these
2. **line_height**: `defaults.line_height` (e.g., 1.4)
3. **accent_foreground**: `defaults.accent_foreground` -- needed for `button.primary_fg` derivation
4. **focus_ring geometry**: `focus_ring_width` and `focus_ring_offset`
5. **icon_sizes**: `defaults.icon_sizes` (toolbar, small, large, dialog, panel) -- currently missing from all community presets
6. **Accessibility defaults**: `text_scaling_factor = 1.0`, `reduce_motion = false`, `high_contrast = false`, `reduce_transparency = false`
7. **Widget fields with no inheritance**: `switch.unchecked_bg`, `switch.track_radius`, `link.underline`, `scrollbar.overlay_mode`, `dialog.button_order` -- these have no derivation source and must be explicit
8. **Sizing fields with no inheritance**: Various widget sizing fields not currently in community presets (e.g., `button.min_width`, `input.padding_vertical`, `checkbox.border_width`, `menu.padding_vertical`, `tooltip.max_width`, `scrollbar.min_thumb_height`, `slider.tick_length`, `progress_bar.min_width`, `tab.padding_vertical`, `toolbar.padding`, `list.padding_vertical`, `dialog.*` sizing, `spinner.*`, `combo_box.*`, `segmented_control.*`, `card.*`, `expander.*`)

### Recommended Approach for Community Presets

Use consistent "community preset defaults" for non-color fields:
- Font: family = "Inter", size = 14.0, weight = 400
- Mono font: family = "JetBrains Mono", size = 14.0
- Line height: 1.4
- Accent foreground: "#ffffff" (light on accent) or theme-appropriate
- Icon sizes: toolbar=24, small=16, large=32, dialog=22, panel=20
- Focus ring: width=2.0, offset=1.0
- Accessibility: text_scaling_factor=1.0, reduce_motion=false, high_contrast=false, reduce_transparency=false
- Overlay mode: false (persistent scrollbars)
- Dialog button order: TrailingAffirmative
- Link underline: true
- Widget sizing: match the "default" preset values

## Open Questions

1. **Should resolve() mutate in-place or return a new ThemeVariant?**
   - What we know: The design doc describes `resolve()` as mutating the variant in place (`pub fn resolve(&mut self)`). This matches the pipeline where resolve() is called, then an app TOML is merged on top, then resolve() is called again.
   - Recommendation: Use `&mut self` to match the pipeline design. The method is idempotent (running it twice produces the same result).

2. **Should ThemeResolutionError be a new Error variant or a separate type?**
   - What we know: The current Error enum has Unsupported, Unavailable, Format, Platform. ThemeResolutionError has structured data (Vec<String>) that doesn't fit the string-based Format variant.
   - Recommendation: Add a `Resolution(ThemeResolutionError)` variant to the existing Error enum. This keeps the crate's Result type unified. validate() returns `crate::Result<ResolvedTheme>`.

3. **How to handle the broken integration tests?**
   - What we know: `native-theme/tests/merge_behavior.rs` references old `ThemeVariant.colors` paths. It does not compile.
   - Recommendation: Update or rewrite integration tests as part of this phase. The merge behavior is still valid; only the field paths changed.

4. **icon_set field on ResolvedTheme -- what default?**
   - What we know: `icon_set` is `Option<String>` on ThemeVariant. Community presets do not set it. For validate(), it needs to become a non-optional `String`.
   - Recommendation: If `icon_set` is None after resolve(), use `"freedesktop"` on Linux, `"sf-symbols"` on macOS, `"segoe-fluent"` on Windows as platform-specific defaults via `system_icon_set()` which already exists. Alternatively, accept that icon_set may remain None until the pipeline phase (Phase 47) wires `from_system()`. For Phase 45, community presets could just not set icon_set and validate() could use a platform fallback.
