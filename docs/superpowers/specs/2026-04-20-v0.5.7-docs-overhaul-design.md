# Design — v0.5.7 Pre-Release Documentation Overhaul

**Status:** approved (2026-04-20), amended (2026-04-21) to preserve existing visual assets, ready for implementation plan
**Trigger:** user wants all project documentation synchronized with v0.5.7 code and optimized for newcomer onboarding before the GitHub release
**Scope:** all 5 crates + workspace-level docs + examples + one supplementary diagram

---

## 1. Goals and success criteria

**Primary goals (in priority order):**

1. **Accuracy** — every type name, API signature, and example in the docs matches the v0.5.7 code. No stale references to renamed types (`ThemeSpec`, `ThemeVariant`, `ResolvedThemeVariant`, `ResolvedThemeDefaults`, the old generic `IconLoader` builder).
2. **Onboarding speed** — a newcomer who lands on the root README reaches a running example within 5 minutes.
3. **No ambiguity / no contradictions** — each term has one definition; crates don't disagree with each other.
4. **Compact** — "beginner-friendly" ≠ "bloated". Reference material stays in rustdoc (docs.rs); READMEs stay concise.

**Audience:** Rust-competent developers who are new to `native-theme`. Rust idioms (`Result`, `Option`, generics, `Cow`, `Arc`) are assumed. Domain concepts (theme resolution, inheritance, presets, connectors) are explained.

**Out of scope:**
- Migration guides or upgrade documents (pre-1.0 policy — no backward-compatibility promises)
- Top-level `examples/` directory (multi-crate workspace; connector-specific examples already exist)
- Rewriting already-good inline `///` docs; only a focused accuracy pass for stale names
- API additions or code changes (pure documentation work)
- Replacing, removing, or regenerating existing visual assets (screenshots, GIFs) — those are preserved as-is; their regeneration pipeline already exists in `scripts/`

## 3. Supplementary diagram — workspace crate relations

Existing screenshots and GIFs (Section 2.0) tell the visual story of what the library produces. The one diagram added in this pass answers a *different* question: "which crate do I depend on?" It is supplementary, not hero.

### 3.1 Specification

**Source:** `docs/assets/crate-relations.mmd` (Mermaid)
**Generated:** `docs/assets/crate-relations.svg`
**Referenced from:** root `README.md` ("How the 5 crates fit together" section), and via `include_str!` also appears in `native-theme/src/lib.rs` crate-level docs. Embedded in the other three crate READMEs too (per Section 2.2 item 2).

Mermaid source (draft — final form during implementation):

```mermaid
flowchart TB
    App([Your Rust App])

    subgraph ws[native-theme workspace]
        direction TB
        NT["<b>native-theme</b><br/>runtime core<br/>Theme, ResolvedTheme, presets"]
        NTG["<b>native-theme-gpui</b><br/>GPUI connector"]
        NTI["<b>native-theme-iced</b><br/>iced connector"]
        NTB["<b>native-theme-build</b><br/>custom icon codegen<br/>(optional, build.rs)"]
        NTD["<b>native-theme-derive</b><br/>proc-macro<br/>(internal)"]
    end

    App -->|depends on| NTG
    App -->|or| NTI
    App -.->|optional| NTB

    NTG --> NT
    NTI --> NT
    NT -.-> NTD

    classDef internal fill:#f0f0f0,stroke:#999,stroke-dasharray: 4 2
    classDef optional fill:#fff8e0,stroke:#c8a000
    class NTD internal
    class NTB optional
```

**Legend rendered below the diagram:**
- Solid arrow = runtime dependency
- Dashed arrow = optional or internal
- Yellow node = optional build-time dependency
- Grey dashed node = library-internal (never import directly)

### 3.2 Data-flow diagram (not included)

An earlier draft considered adding a "Theme → resolve(mode) → ResolvedTheme → render" data-flow diagram. Decision: skip, in favor of a 3-sentence prose explanation in the Core Concepts section. Screenshots already show the *rendered* end state; prose covers the *transformation* concisely without another visual asset to maintain.

### 3.3 Rendering pipeline for the new diagram

- `.mmd` source and `.svg` output are both committed to `docs/assets/` (alongside existing PNGs/GIFs)
- A small helper script `scripts/render-diagrams.sh` regenerates the SVG from Mermaid source via `mmdc` (from `@mermaid-js/mermaid-cli`; prerequisite documented in `CONTRIBUTING.md`)
- `.gitattributes` marks `*.svg` as binary so PR diffs focus on the `.mmd` source (existing PNGs/GIFs are unaffected — they're already binary via extension)
- **Coexists with the existing pipeline** in `scripts/` — does not replace or interact with `generate_screenshots.sh`, `pre-release.sh`, etc.
- No CI wiring — developer runs the script manually after editing the diagram

## 4. Execution approach

**Option C** from brainstorming: pilot + batch.

### 4.1 Pilot (user reviews before batch proceeds)

1. Install `mmdc` setup; write `scripts/render-diagrams.sh`; add `.gitattributes` entry for `*.svg`
2. Author `docs/assets/crate-relations.mmd`; render `docs/assets/crate-relations.svg`
3. Rewrite root `README.md` using Section 2.1 structure; **verify all existing visuals (GIFs, spinners, badges) remain in place and render correctly**
4. Rewrite `native-theme/README.md` using Section 2.2 template; preserve existing spinner GIF block
5. Focused `///` accuracy pass on `native-theme` crate (grep for stale names, update only what's actually stale)
6. Verify `CHANGELOG.md` [0.5.7] entries are accurate and complete against git history since v0.5.6; add any missing entries. Leave the "Unreleased" marker in place — do NOT set a release date autonomously
7. Run `./pre-release-check.sh` — must pass
8. **Stop and request user review.** Atomic commit per deliverable so user can revert any single change surgically.

### 4.2 Batch (proceeds after user approves pilot style)

9. `native-theme-build/README.md` + accuracy pass on its inline docs
10. `native-theme-derive/README.md` (minimal sharpening)
11. `native-theme-gpui/README.md` + accuracy pass; **preserve all 10 per-platform screenshots in their existing layout or a cleaner reorganization (never remove, never regenerate)**
12. `native-theme-iced/README.md` + accuracy pass; **same screenshot preservation policy**
13. `CONTRIBUTING.md` expansion (including reference to `scripts/README.md` for visual regeneration)
14. `SECURITY.md` expansion (with GitHub PVR link)
15. `ROADMAP.md` creation
16. Showcase example comments (`showcase-gpui.rs`, `showcase-iced.rs`)
17. Final verification: full `./pre-release-check.sh`, diff review, list any residual gaps; confirm no visual asset references were broken by README restructuring

## 5. Safety rails

- **No invented content.** Every documented value (spacing, color, font size, OS constant) is cited to `docs/platform-facts.md` + `docs/property-registry.toml` file:line, per project convention.
- **Every code snippet is verified to compile.** Either copied from existing tests, extracted from the showcase examples, or covered by a rustdoc doctest. No free-hand examples.
- **No existing visual asset is removed, moved, or regenerated.** The `docs/assets/*.png`, `*.gif` files are preserved; if a README restructure means a screenshot's markdown reference moves, the image file itself stays put and the reference path is updated.
- **Atomic commits per deliverable.** One doc file update = one commit, with conventional-commit prefix (`docs(scope):`).
- **No `Co-Authored-By` trailers** on any commit (user preference).
- **No tag push, no release, no publish without explicit user approval.** Pilot stop + batch stop are hard checkpoints; CHANGELOG "Unreleased" stays "Unreleased" until user explicitly tags.
- **`./pre-release-check.sh` is the gate.** Runs before pilot review and before final batch completion.

## 7. Non-goals

- No new Rust API work
- No test additions beyond rustdoc doctests that fall out of updated examples
- No refactoring of source code (even if a doc pass surfaces smells; those get logged, not fixed)
- No design-system or theming-content changes (e.g., no new presets, no changed inheritance rules)
- No GitHub Actions / CI changes
- **No regeneration of existing screenshots, theme-switching GIFs, or spinner GIFs.** That pipeline (`scripts/pre-release.sh` and its dependencies) already exists and runs at release time, separately from this doc pass. Running it is the user's call, not this pass's.
- No replacement of existing visuals with new abstractions. Screenshots beat diagrams for "what it looks like"; only one small supplementary diagram is added, answering a structurally different question ("which crate do I depend on?").
