# v0.3.1: Feature Flag Simplification

## Problem

The current feature flags are confusing for users. Common questions:

1. "A lot of platforms are missing (GNOME, XFCE, Cinnamon, Mint, etc.)"
2. "Does a cross-platform binary need all features?"
3. "Why use features instead of auto-detecting the platform?"

## Current State

### Dependency cost per feature

| Feature | Additional crates | Total dep count | Notes |
|---------|-------------------|-----------------|-------|
| *(none)* | -- | 45 | serde + serde_with + toml |
| `kde` | +1 (configparser) | 46 | Pure Rust INI parser, near zero cost. **Not target-gated.** |
| `portal-tokio` | +123 (ashpd, zbus, tokio) | 168 | Heavy: full D-Bus + async stack. **Not target-gated.** |
| `portal-async-io` | +168 (ashpd, zbus, async-io) | 213 | Even heavier. **Not target-gated.** |
| `macos` | objc2 + bindings | target-gated | Only compiles on macOS |
| `windows` | windows crate | **NOT target-gated** | Compiles on all platforms |

### How Linux DEs are handled today

`from_system()` detects `XDG_CURRENT_DESKTOP` and dispatches:

| Desktop | Reader | Feature required |
|---------|--------|------------------|
| KDE | `from_kde()` — parses `~/.config/kdeglobals` | `kde` |
| GNOME | Adwaita preset (sync) / portal (async) | none / `portal-*` |
| Budgie | Same as GNOME | none / `portal-*` |
| XFCE | Adwaita preset | none |
| Cinnamon | Adwaita preset | none |
| MATE | Adwaita preset | none |
| LXQt | Adwaita preset | none |
| Unknown | Adwaita preset | none |

**GNOME, XFCE, Cinnamon, Mint, MATE, LXQt, Budgie do NOT need separate
features.** They all:

- Use GTK themes (CSS), which are impractical to parse
- Support the XDG Desktop Portal for live accent/dark-mode/contrast data
- Get the Adwaita preset as an accurate fallback

The only reason `kde` is a separate feature is the `configparser` dependency.
The reason `portal-*` is a separate feature is the `ashpd` + `zbus` stack
(123-168 extra crates) and the async runtime choice (tokio vs async-io).

### Why features instead of auto-detection?

For macOS and Windows, the platform is known at compile time (`cfg(target_os)`).
Features exist because:

- **macOS**: deps are already target-gated. The `macos` feature just controls
  whether `from_system()` reads NSAppearance or returns `Err(Unsupported)`.
  Without the feature, you can still use presets on macOS.

- **Windows**: deps are NOT target-gated. Enabling the `windows` feature
  on Linux pulls in the `windows` crate unnecessarily.

- **Linux KDE**: `configparser` is 1 crate, not target-gated. Almost zero
  cost so it compiles harmlessly on all platforms, but conceptually wrong.

- **Linux Portal**: `ashpd` + `zbus` is heavy (123+ crates), not
  target-gated, and forces an async runtime choice (tokio vs async-io).
  Enabling `portal-tokio` on macOS compiles zbus+tokio for no reason.

## Proposed Changes

### 1. Target-gate all OS-specific dependencies

Currently only `objc2*` (macOS) and `freedesktop-icons` (Linux) are
target-gated. Three dependencies under `[dependencies]` compile on all
platforms when their feature is enabled:

| Dependency | Feature | Should be gated to |
|------------|---------|-------------------|
| `windows` | `windows` | `cfg(target_os = "windows")` |
| `ashpd` | `portal` | `cfg(target_os = "linux")` |
| `configparser` | `kde` | `cfg(target_os = "linux")` |

Move all three to their respective `[target.'cfg(...)'.dependencies]` sections.
This ensures that enabling `native` on macOS only pulls in objc2, not
zbus+tokio+configparser. Cargo handles this correctly: if a `dep:foo`
feature references a target-gated optional dep, the dep is simply not
compiled on non-matching platforms.

### 2. Add convenience meta-features

```toml
[features]
# One feature per platform for "full native support":
linux = ["kde", "portal-tokio"]
linux-async-io = ["kde", "portal-async-io"]

# Everything — safe to enable unconditionally because OS deps are target-gated:
native = ["linux", "macos", "windows"]
native-async-io = ["linux-async-io", "macos", "windows"]
```

After target-gating all OS deps, a cross-platform app just writes:

```toml
[dependencies]
native-theme = { version = "0.3.1", features = ["native"] }
```

One line. Works on all platforms. Wrong-OS deps compile away.

### 3. Keep individual features for fine-grained control

The meta-features are additive. Users who want minimal deps can still use:

```toml
# Presets only, no OS readers
native-theme = "0.3.1"

# KDE only (1 extra dep), no async
native-theme = { version = "0.3.1", features = ["kde"] }

# Portal with async-io instead of tokio
native-theme = { version = "0.3.1", features = ["kde", "portal-async-io"] }
```

### 4. Document recommended features clearly

Update the README feature table to lead with the meta-features:

| Feature | Use case |
|---------|----------|
| `native` | Full native support on all platforms (tokio) |
| `native-async-io` | Same, but with async-io instead of tokio |
| `linux` | Full Linux support (KDE + GNOME portal via tokio) |
| `macos` | macOS NSAppearance reader |
| `windows` | Windows UISettings reader |
| `kde` | KDE kdeglobals parser only (no async) |
| `portal-tokio` | GNOME portal with tokio |
| `portal-async-io` | GNOME portal with async-io |

### 5. Consider making `kde` default on Linux

`configparser` is a single, pure-Rust crate. It compiles everywhere.
Making it a default feature means `from_system()` reads KDE themes
out of the box with no configuration.

**Trade-off**: every build (including macOS/Windows) compiles configparser.
It's tiny, but conceptually it doesn't belong there. Cargo doesn't support
per-target default features.

**Decision**: keep `kde` optional but include it in `linux` and `native`
meta-features. This is cleaner than a global default.

## What This Doesn't Change

- `from_system()` still auto-detects the DE at runtime. No user code changes.
- `from_system_async()` still uses the portal when available.
- All presets work without any features.
- Individual features remain for fine-grained dependency control.

## Summary

| Before (v0.3.0) | After (v0.3.1) |
|------------------|----------------|
| User must know which features to enable per platform | `features = ["native"]` covers everything |
| `windows`, `ashpd`, `configparser` not target-gated | All OS deps properly target-gated |
| No `linux` shorthand | `linux` = `kde` + `portal-tokio` |
| GNOME/XFCE/etc. users confused about missing features | README explains they're already handled |
