# v0.3.2 Quality Improvements

Milestone targeting code quality, performance, and API hygiene issues identified
during a deep review of the v0.3.1 codebase. No new features — only fixes and
improvements to existing code.

---

## 1. Cache `system_icon_theme()` and `system_is_dark()` results

**Severity:** High (performance)
**Files:** `native-theme/src/model/icons.rs`, `native-theme/src/lib.rs`

### Problem

`system_icon_theme()` spawns subprocesses (`gsettings`, `xfconf-query`) or reads
config files on every call. There is zero caching anywhere in the codebase
(confirmed: no `OnceLock`, `LazyLock`, `once_cell`, or `lazy_static` usage).

`load_freedesktop_icon()` calls `detect_theme()` → `system_icon_theme()` on
**every invocation**. Loading all 42 icon roles at startup means 42 subprocess
spawns just to re-read the same theme name.

`system_is_dark()` has the same pattern — it shells out to `gsettings` on every
call with no caching.

### Solution

Use `std::sync::OnceLock` for both values:

```rust
// In model/icons.rs
use std::sync::OnceLock;

static CACHED_ICON_THEME: OnceLock<String> = OnceLock::new();

pub fn system_icon_theme() -> String {
    CACHED_ICON_THEME.get_or_init(|| detect_linux_icon_theme_inner()).clone()
}
```

```rust
// In lib.rs
static CACHED_IS_DARK: OnceLock<bool> = OnceLock::new();

pub fn system_is_dark() -> bool {
    *CACHED_IS_DARK.get_or_init(|| detect_is_dark_inner())
}
```

**Why `OnceLock` and not `LazyLock`:** The detection logic has side effects
(subprocess spawning, file I/O). `OnceLock::get_or_init` makes the
initialization point explicit and keeps the detection function as a normal `fn`
rather than requiring a closure in a static.

**Trade-off:** Caching means theme changes after process start won't be picked
up. This is acceptable because:
- GUI apps typically check theme at startup and listen for change signals
  separately (via D-Bus portal notifications, etc.)
- The `from_gnome()` async path already handles live changes via portal
- Users who need to re-detect can restart the process
- We can later add a `clear_cache()` function if needed

**Note:** Do NOT cache `from_system()` or `from_linux()` — those return
full `NativeTheme` structs and users may want to re-read (e.g., after a
portal signal says the theme changed). Only cache the cheap-but-repeated
detection helpers.

## 2. Add `pick_variant` method to `NativeTheme`

**Severity:** Low (API duplication)
**Files:** `native-theme/src/model/mod.rs`, both connectors

### Problem

Both `native-theme-iced/src/lib.rs:39-48` and `native-theme-gpui/src/lib.rs:35-41`
contain identical `pick_variant()` implementations:

```rust
pub fn pick_variant(theme: &NativeTheme, is_dark: bool) -> Option<&ThemeVariant> {
    if is_dark {
        theme.dark.as_ref().or(theme.light.as_ref())
    } else {
        theme.light.as_ref().or(theme.dark.as_ref())
    }
}
```

This is a natural method on `NativeTheme` itself. Connector users shouldn't need
to import a connector crate just to pick a variant.

### Solution

Add to `NativeTheme` in `model/mod.rs`:

```rust
impl NativeTheme {
    /// Pick the appropriate variant for the given mode, with cross-fallback.
    ///
    /// When `is_dark` is true, prefers `dark` and falls back to `light`.
    /// When `is_dark` is false, prefers `light` and falls back to `dark`.
    /// Returns `None` only if the theme has no variants at all.
    pub fn pick_variant(&self, is_dark: bool) -> Option<&ThemeVariant> {
        if is_dark {
            self.dark.as_ref().or(self.light.as_ref())
        } else {
            self.light.as_ref().or(self.dark.as_ref())
        }
    }
}
```

Keep the existing free functions in connectors as deprecated thin wrappers for
one release to avoid a breaking change, then remove in v0.4.

## 3. Document `colorize_svg` limitations in iced connector

**Severity:** Medium (correctness for edge cases)
**Files:** `connectors/native-theme-iced/src/icons.rs`

### Problem

`colorize_svg` uses string manipulation to recolor SVG icons:
- `svg_str.replace("currentColor", &hex)` — works for Lucide-style SVGs
- `svg_str.find("<svg")` + inject `fill=` — works for Material-style SVGs

This is fragile for arbitrary SVGs because:
1. `find("<svg")` could match inside a comment or CDATA (unlikely but possible)
2. Injecting before `>` doesn't handle attributes containing `>` in values
3. Multi-color SVGs (e.g., Breeze freedesktop themes with explicit `fill` colors
   on child elements) won't be fully recolored

### Analysis: Is this actually a problem?

For **bundled** Material/Lucide SVGs: No. These are well-structured monochrome
SVGs — the function works perfectly.

For **freedesktop** system theme SVGs: These can be multi-color (Breeze icons
use explicit fill colors for different parts). But calling `to_svg_handle_colored`
on a multi-color icon is already semantically wrong — the user is asking to
force a single color on a multi-color icon. The non-colored `to_svg_handle`
exists for this case.

### Solution

Do **not** add XML parser complexity. Instead:

1. Add doc comments explaining when to use `to_svg_handle_colored` vs
   `to_svg_handle`:

```rust
/// Colorize a **monochrome** SVG icon with the given color.
///
/// Works correctly for bundled icon sets (Material, Lucide) which use
/// `currentColor` or implicit black fills. For multi-color SVGs from
/// freedesktop system themes, use [`to_svg_handle()`] instead to
/// preserve the original icon colors.
```

2. Rename the internal `colorize_svg` function to `colorize_monochrome_svg` to
   make the contract explicit.

## 4. Remove dead `lighten`/`darken` wrappers in gpui derive.rs

**Severity:** Low (dead abstraction)
**Files:** `connectors/native-theme-gpui/src/derive.rs`

### Problem

```rust
pub fn lighten(color: Hsla, factor: f32) -> Hsla {
    color.lighten(factor)  // just calls Colorize trait method
}
pub fn darken(color: Hsla, factor: f32) -> Hsla {
    color.darken(factor)   // just calls Colorize trait method
}
```

These trivially forward to `gpui_component::Colorize` trait methods. They are
only used within `derive.rs` itself (in `active_color`). The `colors.rs` module
imports only `active_color` and `hover_color` — never `lighten`/`darken`.

These are `pub` functions that appear to be public API but add no value.

### Solution

Inline the trait calls in `active_color`:

```rust
pub fn active_color(base: Hsla, is_dark: bool) -> Hsla {
    let factor = if is_dark { 0.2 } else { 0.1 };
    base.darken(factor)
}
```

Remove `lighten` and `darken`. Also remove `with_alpha` if it has no external
callers (it just calls `.alpha()`).

If the functions were intended as public API for connector users, they should
be re-exported from `lib.rs` — but they currently aren't, and direct
`Colorize` trait usage is the idiomatic approach.

## 5. Fix `to_theme` color overwite round-trip in gpui connector

**Severity:** Low (wasted work, not a bug)
**Files:** `connectors/native-theme-gpui/src/lib.rs`

### Problem

```rust
let mut theme = Theme::from(&theme_color);     // (1) sets colors
theme.apply_config(&theme_config.into());       // (2) overwrites colors with defaults
theme.colors = theme_color;                     // (3) restores our colors
```

Step 2 destroys step 1 and step 3 undoes step 2's color damage. The net effect
is that `apply_config` only contributes its non-color side effects (font family,
font size, radius, shadow, mode).

### Solution

Check if gpui-component exposes a way to apply only the non-color fields from
a config. If not, extract the specific fields manually:

```rust
let mut theme = Theme::from(&theme_color);
let config: gpui_component::theme::AppTheme = theme_config.into();
// Apply only non-color fields
theme.mode = config.mode;
// ... fonts, radius, shadow ...
```

If `apply_config` is the only way to set these fields (due to private fields),
then keep the current pattern but add a comment explaining that this is
intentional and why. The current comment says "Note: apply_config sets...
We must restore" which is good but could be more explicit that this is a
known gpui-component API limitation, not a bug.

## 6. Add `#[must_use]` to key public functions

**Severity:** Low (API hygiene)
**Files:** `native-theme/src/lib.rs`, `native-theme/src/model/icons.rs`,
`native-theme/src/model/bundled.rs`, `native-theme/src/model/mod.rs`

### Problem

Several functions return values that should never be silently ignored:

- `from_system()` → `Result<NativeTheme>`
- `from_system_async()` → `Result<NativeTheme>`
- `load_icon()` → `Option<IconData>`
- `system_icon_set()` → `IconSet`
- `system_icon_theme()` → `String`
- `bundled_icon_svg()` → `Option<&[u8]>`
- `bundled_icon_by_name()` → `Option<&[u8]>`
- `NativeTheme::preset()` → `Result<NativeTheme>`
- `NativeTheme::from_toml()` → `Result<NativeTheme>`
- `NativeTheme::from_file()` → `Result<NativeTheme>`
- `NativeTheme::to_toml()` → `Result<String>`
- `NativeTheme::list_presets()` → `&[&str]`

Calling any of these without using the return value is almost certainly a bug.

### Solution

Add `#[must_use]` to all the above. For `Result`-returning functions, use a
descriptive message:

```rust
#[must_use = "this returns the detected theme; it does not apply it"]
pub fn from_system() -> crate::Result<NativeTheme> { ... }
```

Also consider adding `#[must_use]` to the types themselves:
- `NativeTheme` (constructing a theme without using it is likely a bug)
- `IconData` (loading icon data without using it is likely a bug)

## 7. Replace `python3` with `jq` in pre-release-check.sh

**Severity:** Low (tooling)
**Files:** `pre-release-check.sh`

### Problem

Line 128 uses Python 3 to parse `cargo metadata` JSON output:

```bash
WORKSPACE_CRATES=$(cargo metadata --no-deps --format-version 1 2>/dev/null \
    | python3 -c "import sys,json; [print(p['name']) for p in json.load(sys.stdin)['packages']]")
```

This creates an unnecessary runtime dependency on Python 3. Rust projects
typically have `jq` available, and the script already checks for and installs
other cargo tools.

### Solution

Replace with `jq`:

```bash
WORKSPACE_CRATES=$(cargo metadata --no-deps --format-version 1 2>/dev/null \
    | jq -r '.packages[].name')
```

If `jq` might not be installed, add it to the tool-check section at the top of
the script, or use a pure-bash fallback:

```bash
# Fallback if jq is not available
if command -v jq &>/dev/null; then
    WORKSPACE_CRATES=$(cargo metadata --no-deps --format-version 1 | jq -r '.packages[].name')
else
    WORKSPACE_CRATES=$(cargo metadata --no-deps --format-version 1 \
        | grep -o '"name":"[^"]*"' | sed 's/"name":"//;s/"//')
fi
```

## Review notes: issues from the review re-examined in depth

Each issue below was initially flagged, then dismissed, then subjected to a
second deep-reasoning pass with concrete verification (compiled test programs,
real-world file checks, pointer-equality experiments).

### `read_ini_value` prefix match (originally reported as a parsing bug)

**Verdict: CONFIRMED NOT A BUG after exhaustive trace.**

The original concern was that `strip_prefix("Theme")` would match the line
`ThemeExtra=foo`. This is wrong because the parser has a two-stage guard:

```
strip_prefix(key) → remainder → trim_start() → strip_prefix('=')
```

The `strip_prefix('=')` on the remainder acts as a delimiter assertion. Every
dangerous case was tested with a compiled Rust program:

| Line                    | Key     | strip_prefix(key) | After trim | strip_prefix('=') | Result    |
|-------------------------|---------|--------------------|------------|-------------------|-----------|
| `Theme=breeze`          | `Theme` | `=breeze`          | `=breeze`  | `breeze`          | Match     |
| `ThemeExtra=foo`        | `Theme` | `Extra=foo`        | `Extra=foo`| None              | No match  |
| `ThemeColor=red`        | `Theme` | `Color=red`        | `Color=red`| None              | No match  |
| `Theme = breeze`        | `Theme` | ` = breeze`        | `= breeze` | ` breeze`→`breeze`| Match     |
| `# Theme=old`           | `Theme` | None (starts `#`)  | —          | —                 | No match  |
| `Theme=`                | `Theme` | `=`                | `=`        | `` (empty)        | No match  |

Additional edge cases verified: inline `#` in values (KConfig does not support
inline comments, so `Theme=breeze # note` correctly returns `breeze # note`),
duplicate sections (parser re-enters the section and finds the first matching
key across all instances), comment lines starting with `;` or `#` (rejected
because `strip_prefix(key)` fails).

Real-world validation: checked `/home/tibi/.config/kdedefaults/kdeglobals` —
the `[Icons]` section contains only `Theme=breeze-dark`, no prefix-colliding
keys exist in KDE's actual config format.

### `ThemeVariant::is_empty()` MSRV concern with `is_none_or`

**Verdict: CONFIRMED NOT AN ISSUE — impossible to trigger.**

Three constraints make this completely safe:

1. `Option::is_none_or` stabilized in **Rust 1.82.0**
2. Edition 2024 requires **Rust >= 1.85.0** (compiler floor)
3. Workspace declares `rust-version = "1.94.0"` (explicit MSRV)

The only way this could become a problem is lowering the MSRV below 1.82, which
would also require downgrading from edition 2024 (impossible — the edition sets
a higher floor at 1.85). The constraints form a locked chain: edition 2024 →
1.85 minimum → `is_none_or` available (1.82). No action needed.

### Binary size from `include_bytes!` duplication

**Verdict: CONFIRMED ACCEPTABLE — compiler deduplicates, verified empirically.**

The concern was that `include_bytes!("../../icons/material/delete.svg")` used in
three different match arms (in `material_svg` for `TrashEmpty`/`TrashFull` and in
`material_svg_by_name` for `"delete"`) would embed the file three times.

Verification: compiled a test program with two `static` bindings pointing to
the same file via `include_bytes!`. Pointer comparison shows **identical
addresses in both debug and release builds**. The Rust compiler deduplicates
at the `include_bytes!` expansion level — not even relying on linker ICF.

```
=== Debug build ===
Same pointer: true
=== Release build ===
Same pointer: true
```

Additionally, existing test assertions enforce size budgets:
- `material_icons_total_size_under_200kb` (checks < 400KB)
- `lucide_icons_total_size_under_100kb` (checks < 200KB)

Minor note: the test function names are misleading (say "200kb"/"100kb" but
assert 400KB/200KB). This is a documentation inconsistency, not a bug, and
not worth a milestone item.

### Test verbosity (derive trait tests)

**Verdict: CONFIRMED ACCEPTABLE — zero risk, marginal value, no cost.**

Tests like `icon_role_derives_copy_clone` and `icon_role_derives_hash` verify
behavior guaranteed by `#[derive(Copy, Clone, Hash)]`. Removing a derive would
cause compilation errors at all call sites long before a test would catch it.

However, these tests are:
- Zero-cost to maintain (no code changes trigger updates)
- Instantaneous to run (no I/O, no allocations)
- Serve as executable documentation of the type's capabilities
- Would catch a hypothetical manual trait impl that replaces a derive

The trade-off (mild test noise vs. zero cost) does not warrant action.
