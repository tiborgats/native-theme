# Generate validate.rs extraction with a derive macro

Status: Not started
Date: 2026-04-09

---

## Problem

`resolve/validate.rs` is 2,196 lines. The bulk of it (~1,800 lines) is
hand-written field extraction boilerplate inside `ThemeVariant::validate()`:

```rust
// This pattern repeats ~150 times:
let button_background = require(
    &self.button.background_color,
    "button.background_color",
    &mut missing,
);
```

Each new field or widget requires adding:
1. A `require()` / `require_font_opt()` / `require_border()` call
2. The correct string path (must match the struct field name — no compiler check)
3. A range check if the field is numeric
4. The field in the `ResolvedThemeVariant` construction block

The string paths are hand-typed and never validated against the actual struct
field names. A typo in `"button.backgrund_color"` would produce an unhelpful
error message while compiling fine. The 3-step "add field" process documented
at `validate.rs:292-297` is an admission that this code is fragile.

---

## Proposed solution

A derive macro in `native-theme-build` that generates the extraction and
construction code from annotations on the Resolved widget structs.

### Approach: annotate ResolvedXxxTheme structs

Today `define_widget_pair!` generates both the Option-based widget struct
and the Resolved struct. The derive macro would annotate the Resolved
struct to describe how each field is extracted from its Option-based source:

```rust
// In model/resolved.rs (or generated by define_widget_pair!):
#[derive(ValidateExtract)]
#[validate(source = "ButtonTheme", prefix = "button")]
pub struct ResolvedButtonTheme {
    #[validate(require)]
    pub background_color: Rgba,

    #[validate(require)]
    pub primary_background: Rgba,

    #[validate(require)]
    pub primary_text_color: Rgba,

    #[validate(require)]
    pub min_width: f32,

    #[validate(require)]
    pub min_height: f32,

    #[validate(require)]
    pub icon_text_gap: f32,

    #[validate(require)]
    pub disabled_opacity: f32,

    #[validate(font)]
    pub font: ResolvedFontSpec,

    #[validate(border)]
    pub border: ResolvedBorderSpec,
}
```

The macro generates:
```rust
impl ResolvedButtonTheme {
    pub(crate) fn extract(
        source: &ButtonTheme,
        dpi: f32,
        missing: &mut Vec<String>,
    ) -> Self {
        let background_color = require(
            &source.background_color,
            "button.background_color",
            missing,
        );
        let primary_background = require(
            &source.primary_background,
            "button.primary_background",
            missing,
        );
        // ... etc for all fields ...
        let font = require_font_opt(&source.font, "button.font", dpi, missing);
        let border = require_border(&source.border, "button.border", missing);
        Self {
            background_color,
            primary_background,
            primary_text_color,
            min_width,
            min_height,
            icon_text_gap,
            disabled_opacity,
            font,
            border,
        }
    }
}
```

Then `validate()` shrinks to:
```rust
pub fn validate(&self) -> crate::Result<ResolvedThemeVariant> {
    let mut missing = Vec::new();
    let dpi = self.defaults.font_dpi.unwrap_or(DEFAULT_FONT_DPI);

    let defaults = ResolvedThemeDefaults::extract(&self.defaults, dpi, &mut missing);
    let text_scale = ResolvedTextScale::extract(&self.text_scale, dpi, &mut missing);
    let window = ResolvedWindowTheme::extract(&self.window, dpi, &mut missing);
    let button = ResolvedButtonTheme::extract(&self.button, dpi, &mut missing);
    // ... one line per widget ...

    // Range checks (kept manual — these are semantic, not structural)
    check_range_f32(defaults.disabled_opacity, 0.0, 1.0,
                    "defaults.disabled_opacity", &mut missing);
    // ... etc ...

    if missing.is_empty() {
        Ok(ResolvedThemeVariant { defaults, text_scale, window, button, ... })
    } else {
        Err(crate::Error::Resolution(ThemeResolutionError { missing_fields: missing }))
    }
}
```

### Validation attribute variants

| Attribute | Source field type | Extraction method |
|-----------|-----------------|-------------------|
| `#[validate(require)]` | `Option<T>` | `require()` — records path if None |
| `#[validate(font)]` | `FontSpec` (nested, not Option) | `require_font()` — extracts sub-fields |
| `#[validate(font_opt)]` | `Option<FontSpec>` | `require_font_opt()` — records path if None, then sub-fields |
| `#[validate(border)]` | `Option<BorderSpec>` | `require_border()` — inherited sub-fields required |
| `#[validate(border_partial)]` | `Option<BorderSpec>` | `require_border_partial()` — only color+line_width required |
| `#[validate(border_optional)]` | `Option<BorderSpec>` | `border_all_optional()` — no missing errors, all default |
| `#[validate(text_scale_entry)]` | `Option<TextScaleEntry>` | `require_text_scale_entry()` |
| `#[validate(default)]` | `Option<T>` | `unwrap_or_default()` — no missing error |
| `#[validate(skip)]` | `T` | Direct copy, no validation |

### Special cases

**ThemeDefaults** has unique extraction patterns:
- `font_dpi` is extracted before everything else and passed as parameter
- `line_height` is a standalone f32
- Some fields like `icon_theme`, `icon_set`, `dialog_button_order` are
  optional (use `unwrap_or_default`)

These are handled by specific attribute variants or by keeping a small
manual section for defaults only.

**Range checks** remain manual in `validate()`. They are semantic
constraints (opacity 0..=1, weight 100..=900, min <= max) that don't
map cleanly to field-level annotations without over-engineering the
attribute DSL. Keeping ~50 lines of range checks manual is acceptable.

## Alternatives considered

### A. Macro-free refactor (extract per-widget functions by hand)

Move each widget's extraction into `impl ResolvedXxxTheme { fn extract(...) }`,
written manually.

Pros: No proc-macro dependency, immediate.
Cons: Still hand-typed paths, still fragile, still ~1,200 lines of boilerplate.
Saves structure but not maintenance burden.

### B. Compile-time path validation via const assertions

Keep hand-written extraction but use a test or const fn to verify that
every path string in `validate()` matches a real field name on the source struct.

Pros: Catches typos.
Cons: Doesn't reduce boilerplate. Fragile string matching. Half-measure.

### C. Generate from define_widget_pair! directly

Extend `define_widget_pair!` to also emit the extraction code, since it
already knows all fields.

Pros: Single source of truth (the macro invocation).
Cons: `define_widget_pair!` is already complex (model/widgets/mod.rs:48-96).
Adding extraction logic makes it harder to understand. Proc-macro in
native-theme-build is more readable and independently testable.

## Impact

| Before | After |
|--------|-------|
| validate.rs: ~2,196 lines | validate.rs: ~250 lines (range checks + construction) |
| New field: edit 3 places in validate.rs | New field: add field to Resolved struct with attribute |
| String path typo: silent wrong error msg | String path: generated from field name — always correct |
| native-theme-build: icon codegen only | native-theme-build: icon codegen + ValidateExtract derive |

## Risk

Medium. The proc-macro must correctly handle all 6 extraction variants
and produce identical behavior to the hand-written code. Mitigation:
the existing test suite (`resolve/tests.rs`, 2,152 lines) validates
every extraction path and error message. Run the full suite before/after.

## Dependencies

- `native-theme-build` already exists and is a build dependency
- The `syn` and `quote` crates would need to be added to native-theme-build
  (they are standard proc-macro dependencies)
- `native-theme-build` currently uses `proc-macro = false` (it's a regular
  library, not a proc-macro crate). The derive macro would need either:
  - A new `native-theme-derive` crate (proc-macro = true), or
  - Converting native-theme-build to a proc-macro crate (breaking change
    for existing build.rs consumers)
  Recommendation: new `native-theme-derive` crate to avoid disruption.

## Verification

- `cargo test --features native` passes
- All error messages from `validate()` are identical (compare test output)
- `cargo expand` on a widget struct shows correct extraction code
- `cargo clippy --features native -- -Dwarnings` passes
