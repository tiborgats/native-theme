# Generate validate.rs extraction with a derive macro

Status: Not started
Date: 2026-04-09

---

## Problem

`resolve/validate.rs` is 2,196 lines. The bulk of it (~1,800 lines) is
hand-written field extraction boilerplate inside `ThemeVariant::validate()`:

```rust
// This pattern repeats ~150 times:
let button_background = require(
    &self.button.background_color,
    "button.background_color",
    &mut missing,
);
```

Each new field or widget requires adding:
1. A `require()` / `require_font_opt()` / `require_border()` call
2. The correct string path (must match the struct field name — no compiler check)
3. A range check if the field is numeric
4. The field in the `ResolvedThemeVariant` construction block

The string paths are hand-typed and never validated against the actual struct
field names. A typo in `"button.backgrund_color"` would produce an unhelpful
error message while compiling fine. The 3-step "add field" process documented
at `validate.rs:292-297` is an admission that this code is fragile.

---

## Options

### Option A: Hand-written per-widget extract() functions

Move each widget's extraction into `impl ResolvedXxxTheme { fn extract(...) }`
written manually. The `validate()` body shrinks to ~80 lines (one `extract()`
call per widget + range checks + construction), but the extraction code
itself (~1,200 lines) still exists, just distributed across 25 impl blocks.

**Pros:**
- No new dependencies. No proc-macro. Immediate.
- Each widget's extraction is self-contained and testable.
- Field paths are still hand-typed, but now colocated with the struct they
  extract from — easier to audit.

**Cons:**
- String paths are still unchecked — typos still compile fine.
- Total boilerplate only drops modestly (~2,196 → ~1,300 lines across files).
- Adding a field still requires manual work in the extract function.

### Option B: Extend define_widget_pair! to emit extract()

`define_widget_pair!` (model/widgets/mod.rs:48-96) already knows every
field name and type for both the Option struct and the Resolved struct.
It could emit an `extract()` method alongside the structs it generates.

The macro currently produces:
- `ButtonTheme` (Option-based)
- `ResolvedButtonTheme` (all fields required)
- `impl_merge!(ButtonTheme { ... })`

It would additionally produce:
- `impl ResolvedButtonTheme { fn extract(source: &ButtonTheme, dpi: f32, missing: &mut Vec<String>) -> Self { ... } }`

Field categories already exist in the macro (`option`, `soft_option`,
`optional_nested`) and map directly to extraction strategies:
- `option` fields → `require(&source.field, "prefix.field", missing)`
- `soft_option` fields → `source.field` (keep as Option, no extraction)
- `optional_nested` with `[FontSpec, ResolvedFontSpec]` → `require_font_opt()`
- `optional_nested` with `[BorderSpec, ResolvedBorderSpec]` → `require_border()`

**Pros:**
- Single source of truth: field list defines struct + merge + extraction.
- Zero new crates, zero new dependencies.
- String paths are generated from field names — typos impossible.
- Adding a field to the macro automatically generates its extraction.

**Cons:**
- `define_widget_pair!` is already the most complex macro in the crate (50 lines).
  Adding extraction logic roughly doubles it. Debugging macro expansion errors
  is harder than debugging plain functions.
- Border extraction has 3 variants (full / partial / all-optional) and font
  extraction has 2 variants (nested FontSpec / Option<FontSpec>). The macro
  needs to distinguish these — either via new field category keywords or
  per-field attributes. This adds DSL complexity.
- `ThemeDefaults` and `TextScale` are NOT generated by `define_widget_pair!` —
  they have unique extraction patterns. Their extraction stays manual or needs
  a separate macro.
- Macro errors show expanded code positions, not source positions — harder to
  debug when extraction logic fails.

### Option C: Proc-macro derive in new native-theme-derive crate

A `#[derive(ValidateExtract)]` proc-macro annotates the Resolved structs:

```rust
#[derive(ValidateExtract)]
#[validate(source = "ButtonTheme", prefix = "button")]
pub struct ResolvedButtonTheme {
    #[validate(require)]
    pub background_color: Rgba,
    #[validate(font)]
    pub font: ResolvedFontSpec,
    #[validate(border)]
    pub border: ResolvedBorderSpec,
}
```

Generates the same `extract()` method as Option B, but in a proc-macro crate.

**Attribute variants needed:**

| Attribute | Source type | Extraction |
|-----------|-----------|------------|
| `require` | `Option<T>` | `require()` — records path if None |
| `font` | `FontSpec` (nested) | `require_font()` — sub-fields |
| `font_opt` | `Option<FontSpec>` | `require_font_opt()` |
| `border` | `Option<BorderSpec>` | `require_border()` — inherited fields required |
| `border_partial` | `Option<BorderSpec>` | `require_border_partial()` — color+line_width only |
| `border_optional` | `Option<BorderSpec>` | `border_all_optional()` — all default |
| `text_scale_entry` | `Option<TextScaleEntry>` | `require_text_scale_entry()` |
| `default` | `Option<T>` | `unwrap_or_default()` — no error |
| `skip` | `T` | Direct copy |

**Pros:**
- Cleanest result: extraction is declarative, colocated with the Resolved struct.
- Full compiler support for attributes — IDE autocomplete, refactoring.
- String paths generated from field names — typos impossible.
- Independent from `define_widget_pair!` — no added complexity there.
- Can handle ThemeDefaults and TextScale with the same derive.

**Cons:**
- Requires a new `native-theme-derive` crate (proc-macro = true).
  `native-theme-build` is a regular library (not proc-macro) used in build.rs
  by consumers for icon codegen — converting it to proc-macro would break that.
- Adds `syn` + `quote` dependencies to the workspace (~25K lines of code,
  ~2-3s compile time on first build). Already in the workspace via
  `serde_derive`, but still adds to proc-macro compilation.
- The 9 attribute variants are a small DSL to maintain.
- Macro expansion is opaque — `cargo expand` needed to debug generated code.

### Option D: Compile-time path validation only

Keep hand-written extraction but add a test that reflects over field names
(via FIELD_NAMES const arrays already present on widget structs) to verify
every string path in `validate()` corresponds to a real field.

**Pros:**
- Catches the specific bug (path string typos) with minimal change.
- No new crates, no macro complexity.

**Cons:**
- Doesn't reduce boilerplate at all — still 2,196 lines.
- String matching is fragile (must account for prefixes, nested paths).
- Half-measure: addresses correctness but not maintainability.

---

## Recommendation: Option B

Option B (extend `define_widget_pair!`) is the best trade-off:

1. **No new crate.** Option C's `native-theme-derive` adds workspace
   complexity and proc-macro compile overhead for a problem that's
   solvable within the existing macro system.

2. **Single source of truth.** The widget field list already appears in
   exactly one place (`define_widget_pair!` invocation). Extraction should
   derive from that same source, not from a separate annotation.

3. **The DSL complexity is manageable.** The border variant distinction
   (full / partial / optional) can be expressed with a new field category
   keyword in the macro, e.g., `border_full { border }` vs
   `border_optional { border }`. This adds ~30 lines to the macro, not
   the feared doubling — the core extraction per-field is a 3-line pattern.

4. **ThemeDefaults stays manual.** ThemeDefaults has ~40 fields with
   unique extraction patterns (font_dpi bootstrapping, nested border
   sub-fields, standalone line_height). Its extraction stays as a
   hand-written function (~200 lines). This is acceptable: ThemeDefaults
   changes rarely, and it's one struct, not 25.

5. **Incremental path.** Start by extending `define_widget_pair!` for the
   simplest widgets (those with only `option` fields and one `optional_nested`
   border/font). Validate that the generated code matches the hand-written
   version. Then convert more widgets. The hand-written code in validate.rs
   serves as the reference during migration.

**validate.rs after Option B:** ~350 lines (ThemeDefaults manual extraction +
range checks + ResolvedThemeVariant construction + one extract() call per widget).

---

## Risk

Low-medium. The macro extension is the risky part — `define_widget_pair!`
is already the most complex macro in the crate. Mitigation: the existing
test suite (resolve/tests.rs, 2,152 lines) validates every extraction
path and error message. Run full suite before/after each widget migration.
