## 1. Screenshots and Animated GIFs

The project is a **visual** theming library with zero visual assets in any
README. Every popular GUI crate (iced, egui, ratatui, catppuccin) leads with
screenshots or GIFs. For a theming crate, visuals are arguably even more
important than for a framework — people need to see what they're getting.

### What to capture

- [ ] Screenshots of the iced showcase with 3-4 different presets
      (e.g., Dracula dark, Nord light, Catppuccin Mocha, macOS Sonoma)
- [ ] Screenshots of the gpui showcase with the same presets
- [ ] Animated GIF of the spinner animations running (the v0.4.0 headline
      feature — must be visible in the README)
- [ ] GIF of live theme switching in one of the showcases
- [ ] Add an `assets/` or `docs/images/` directory for these files

### Where to embed them

- [ ] Root `README.md` — hero section with best showcase screenshots
- [ ] `native-theme/README.md` — spinner GIF in the "Animated Icons" section
- [ ] `connectors/native-theme-gpui/README.md` — gpui showcase screenshot
- [ ] `connectors/native-theme-iced/README.md` — iced showcase screenshot

### How to generate them (automation strategy)

Add a `--screenshot` flag to the existing showcase examples. When passed, the
example renders its views headlessly (or captures after launch), saves PNGs to
`assets/`, and exits. This way screenshots always match the real example code
— no separate binary, no duplicated view logic, single source of truth.

**Iced showcase `--screenshot` mode:**

**Verify first:** Check whether `iced_test` (with a headless `Simulator` using
tiny-skia) is available for iced 0.14. If it is:
1. Skip `iced::application(...).run()`, use `iced_test::Simulator` instead
2. For each preset (or a curated subset), construct the view and render
3. Save RGBA output as PNG via the `image` crate
4. For the theme-switching GIF, render multiple presets in sequence and combine
   frames with the `gif` or `gifski` crate

If `iced_test` is not available, fall back to launching the app under Xvfb on
Linux CI (or natively on macOS/Windows runners) and capturing with a tool like
`scrot` or `screencapture`.

**gpui showcase `--screenshot` mode:**

gpui requires Vulkan/Metal (no software renderer). When `--screenshot` is
passed:
- Launch the app normally, render the desired views, use gpui's internal
  render-to-image APIs if available, save and exit
- **Linux CI**: Needs Mesa lavapipe (`jakoch/install-vulkan-sdk-action` with
  `install_lavapipe: true`) + Xvfb. Fragile but possible.
- **macOS/Windows CI**: Runners have real displays, so the app can launch,
  capture its own framebuffer, and exit — no external screenshot tool needed.
- **Practical fallback**: If gpui headless proves too fragile in CI, take gpui
  screenshots manually on your KDE machine. Automate later if gpui gains a
  software backend.

**Spinner GIF — fully automatable, no display needed:**

The spinner animation frames are already SVG files in `native-theme/animations/`.
Add a `--gif` flag (or a separate small script in `scripts/`) that:
1. Rasterizes each frame SVG with `resvg` (already a dependency via
   `svg-rasterize`)
2. Encodes into an animated GIF using the `gif` crate (simple, lightweight) or
   `gifski` (better color quantization for smoother results)
3. Outputs files like `assets/spinner-material.gif`, `assets/spinner-macos.gif`

Pure computation — no display server, no GPU, works anywhere.

**Cross-platform native theme screenshots via CI:**

GitHub Actions provides real Windows and macOS runners with genuine system
themes. The `--screenshot` flag lets the same example binary produce authentic
native-theme screenshots on each platform.

| Platform | CI Runner | `from_system()` result | Notes |
|----------|-----------|------------------------|-------|
| Linux KDE | Your machine | Real KDE theme | Local run |
| Linux GNOME | `ubuntu-latest` | Adwaita preset fallback | No D-Bus portal on CI — but acceptable stock GNOME representation |
| Windows | `windows-latest` | Real UISettings data | Genuine Windows accent colors, dark/light mode |
| macOS | `macos-latest` | Real NSAppearance data | Genuine macOS appearance, accent color |

Example CI workflow triggered on release tags:
```yaml
jobs:
  screenshots:
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - run: cargo run -p native-theme-iced --example showcase -- --screenshot
      - uses: actions/upload-artifact@v4
        with:
          name: screenshots-${{ matrix.os }}
          path: assets/*.png
```

For a "real" GNOME screenshot with portal accent colors, use a QEMU/KVM VM
with Fedora/Ubuntu GNOME on your KDE machine (`virt-manager`). CI runners are
server images with no desktop environment, so `from_gnome()` can't reach a
portal there.
