# v0.5.7 Gaps — Closing Every Remaining Item Before Ship

This document enumerates every item from the three v0.5.7 todo docs that was
either not implemented, implemented partially, or explicitly deferred to a
later milestone. Under the directive *"no backward compat; I want the perfect
API — nothing half-baked, nothing deferred"*, each gap is to be closed before
v0.5.7 ships.

The v0.5.7 milestone audit of 2026-04-15 found 55/55 requirements satisfied
and zero critical blockers. A six-agent independent re-audit on 2026-04-19
(run against the current tree, not the phase verification reports) confirmed
that audit but surfaced the list below: a handful of intentional deviations,
one partially-complete codegen step, three cross-doc architectural items
that were tagged "defer to v0.5.8", and two trivial adds. None are bugs.
All are polish.

## G1 · Remove `Rgba`'s `Default` implementation (P1)

**Source:** doc 1 §16 Recommended (lines 1955-1958):
> Remove `#[derive(Default)]` from `Rgba`. Rename `Rgba::rgba(...)` →
> `Rgba::new(...)`. Delete `Rgba::to_f32_tuple`.

**Status:** two of the three items done. The `Default` derive was removed
but immediately replaced by a manual impl:

```rust
// native-theme/src/color.rs:54-61
/// Returns [`Rgba::TRANSPARENT`] `(0, 0, 0, 0)`.
///
/// Prefer [`Rgba::TRANSPARENT`] or [`Rgba::rgb`]/[`Rgba::new`] for clarity.
impl Default for Rgba {
    fn default() -> Self {
        Self::TRANSPARENT
    }
}
```

The manual impl preserves the exact footgun §16 called out: a theme library
where `Rgba::default()` returns an invisible colour. Phase 90 verification
flagged this as a gap and accepted it because ~30 dependent types
(`ResolvedBorderSpec`, `ResolvedFontSpec`, `ResolvedDefaults`, all 26
`Resolved*Theme` structs) derive `Default` and thus transitively require
`Rgba: Default`.

**Evidence of real dependencies:**
- `native-theme/src/model/border.rs:114` — `pub color: Rgba` in
  `ResolvedBorderSpec` with `#[derive(Default)]`
- `native-theme/src/model/font.rs:244` — `pub color: Rgba` in `ResolvedFontSpec`
- `native-theme/src/model/resolved.rs:77-91` — 8 non-`Option` `Rgba`
  fields on `ResolvedDefaults` (background_color, text_color, accent_color,
  accent_text_color, surface_color, muted_color, shadow_color, link_color)
- All 26 resolved widget structs derive `Default` via `ThemeWidget`.

**Zero existing callers of `Rgba::default()`:** grep across
`native-theme/src/` + `connectors/` finds zero `Rgba::default()` calls.
Every `Rgba` comes from a TOML-loaded preset, a `from_f32` / `from_str` /
`rgb` / `new` call, or a constant (`TRANSPARENT`/`BLACK`/`WHITE`).

**Required fix:** drop the manual `impl Default` from `color.rs:57-61`
and simultaneously drop `Default` from every `Resolved*` struct that
transitively required it. Because no caller invokes `Rgba::default()`
or any resolved-struct `::default()` (resolved structs are produced by
`resolve()`, never default-constructed), the removal is mechanical.

**Concrete plan:**
1. Remove `Default` from the `derive` list on all `Resolved*` widget
   structs generated by `ThemeWidget` (requires `native-theme-derive/src/gen_structs.rs`
   to drop `Default` from its `derive` output — grep for `quote! { #[derive(...)] }`).
2. Remove `Default` from hand-written `ResolvedBorderSpec` (`border.rs:112`),
   `ResolvedFontSpec` (`font.rs:234`), `ResolvedDefaults` (`resolved.rs:31`),
   `ResolvedTheme` (`resolved.rs`), `ResolvedLayoutTheme`.
3. Remove `impl Default for Rgba` from `color.rs:57-61`.
4. Grep-audit: `cargo build` to surface any caller that silently relied on
   the trait bound (expect zero).
5. Update `color.rs:54-56` doc comment to simply describe `TRANSPARENT`.

**Bundled with:** nothing — standalone four-file commit.

**Confidence:** high. Previously blocked by the unverified belief that
Resolved structs need `Default`; the grep audit confirms they do not.

## G4 · `icon_theme` on `Theme` with optional per-variant override (P1)

**Source:** doc 1 §20 Recommended (Option B, lines 2208-2221):
> Move `icon_set` and `icon_theme` to `Theme`.

**Status:** `icon_set` moved (`model/mod.rs:281`). `icon_theme` **stayed
on `ThemeDefaults`** (`model/defaults.rs:125`) per the Phase 80 decision
documented in place:

> This field lives on `ThemeDefaults` (per-variant) rather than on `Theme`
> (shared) because some platforms use different icon themes per color
> mode. For example, KDE Plasma uses `breeze` for light mode and
> `breeze-dark` for dark mode.

That rationale is real. KDE Plasma does ship `breeze` + `breeze-dark` as
two distinct icon themes, and the KDE reader at `kde/mod.rs` populates
each variant's `icon_theme` independently. Option B's "move to Theme,
add override field later" reverses that already-shipped reality.

This is **not** a pure deviation — it is the case that Option B as
written was ill-informed. The merge-review noted this implicitly
(§20 Option C: *"keep `ThemeVariant::icon_set_override: Option<IconSet>`
for the rare case"*) but §20 recommended B on YAGNI grounds.

**Required fix** — adopt §20 Option C, now that the use case is
known-real:

1. Add `icon_theme: Option<Cow<'static, str>>` to `Theme`
   (`model/mod.rs` alongside the existing `icon_set` field).
2. Keep `icon_theme: Option<Cow<'static, str>>` on `ThemeDefaults`
   as the per-variant override.
3. At resolve time (`resolve/mod.rs` near the defaults extraction),
   pick the per-variant value if present, else fall back to the
   `Theme`-level value, else fall back to `IconSet::name()`.
4. Document precedence at both field sites.
5. Update all 16 bundled preset TOMLs that set `icon_theme` to declare
   it at the top level when it's shared (e.g. `adwaita.toml`) and only
   per-variant when it actually differs (e.g. `kde-breeze.toml`).

**Confidence:** high on direction. Medium on preset-migration scope
(need to read all 16 TOMLs to identify which have identical light/dark
`icon_theme` values that can move up).

## G5 · Drive non-widget `FIELD_NAMES` from the derive macro (P1)

**Source:** doc 2 §I5 Recommended (registry-driven lint_toml, lines
2561-2591). Widget `FIELD_NAMES` now come from
`inventory::submit!(WidgetFieldInfo { ... })` emitted by
`native-theme-derive` (`lib.rs:87-111`). Non-widget structs still hand-author:

```
native-theme/src/model/font.rs:171     FontSpec::FIELD_NAMES         (6 fields)
native-theme/src/model/font.rs:268     ResolvedFontSpec::FIELD_NAMES (~14 fields)
native-theme/src/model/font.rs:372     TextScale::FIELD_NAMES        (4 fields)
native-theme/src/model/border.rs:44    DefaultsBorderSpec::FIELD_NAMES
native-theme/src/model/border.rs:93    WidgetBorderSpec::FIELD_NAMES
native-theme/src/model/defaults.rs:130 ThemeDefaults::FIELD_NAMES    (~35 fields)
native-theme/src/model/icon_sizes.rs:32 IconSizes::FIELD_NAMES       (~6 fields)
```

These constants feed the same `lint_toml` call in `model/mod.rs:584-735`.
Any field rename must be mirrored in the constant, and drift is
unenforced by the compiler (only caught by failing a test). §I5 flagged
this as "~174 field-name repetitions".

**Required fix:** one of two directions —

- **Direction A (minimum):** add a `#[derive(FieldNames)]` macro to
  `native-theme-derive` that inspects a plain struct (not `ThemeWidget`)
  and emits `pub const FIELD_NAMES: &[&str]` derived from the struct's
  serde field names. Apply to all 7 non-widget structs above. Delete
  the hand-authored constants.

- **Direction B (preferred, since we already have inventory):** extend
  the existing `ThemeWidget` derive — or add a sister macro
  `#[derive(ThemeFields)]` — to emit the same `inventory::submit!`
  pattern for non-widget structs. Merge widget and non-widget into a
  single `inventory::iter::<FieldInfo>()` consumed by `lint_toml`. This
  removes the FIELD_NAMES constants entirely.

Direction B is doc 2 §I5's intended endgame. Pick B.

**Concrete plan (Direction B):**
1. Add `#[derive(ThemeFields)]` to `native-theme-derive` emitting
   `inventory::submit!(FieldInfo { struct_name, field_names })`.
2. Apply to `FontSpec`, `ResolvedFontSpec`, `TextScale`,
   `DefaultsBorderSpec`, `WidgetBorderSpec`, `ThemeDefaults`,
   `IconSizes` (7 struct annotations).
3. Rewrite `lint_toml` in `model/mod.rs:584-735` to consume the
   unified inventory instead of indexing into seven named constants.
4. Delete all 7 hand-authored `FIELD_NAMES` constants.
5. Update the drift-guard test so a renamed field fails codegen
   rather than needing a manual mirror update.

**Confidence:** high. Mechanism already exists; this is application.

3. `native-theme-derive/src/gen_inherit.rs` (already exists at 66 LOC)

5. Delete the hand-written rules in `inheritance.rs` that are now

6. Keep the drift test but flip its direction: now it asserts the

This is the "B6 codegen migration" that L.5 sequenced as Unit 9 and

**Confidence:** medium on scope (attribute syntax needs to express

## G7 · Introduce `ResolutionContext` with `from_system()` / `for_tests()` (P2)

**Source:** doc 2 §J.2 refinement on B5 (lines 2668-2708):
> `impl ResolutionContext { pub fn from_system(), pub fn for_tests() }`
> `impl Theme { pub fn resolve(self, ctx: Option<&ResolutionContext>) -> Result<...> }`

**Status:** not present. Grep for `ResolutionContext` across
`native-theme/src/` returns zero hits.

Instead, `font_dpi` was threaded as an explicit parameter on
`ThemeMode::into_resolved(font_dpi: Option<f32>)` (`resolve/mod.rs:123`).
`OverlaySource::font_dpi` (`lib.rs:335`) carries it through the replay
path. This works, but:

1. Adding a second resolution-time input (e.g. timezone for date-format
   tokens, locale for number-format tokens) means threading another
   parameter through the same signatures.
2. Tests that want deterministic DPI must pass `Some(96.0)` literally
   at every call site; there is no named `for_tests()` constructor
   signalling intent.
3. The merge-review at J.2 specifically flagged "runtime-detected types
   should not have silent `Default` implementations". Today, tests call
   `into_resolved(None)` which silently falls back to 96 DPI — exactly
   the footgun J.2 wanted to prevent with the explicit `for_tests()`
   naming.

**Required fix:**

1. Create `native-theme/src/resolve/context.rs`:
   ```rust
   #[derive(Clone, Debug)]
   pub struct ResolutionContext {
       pub font_dpi: f32,
       pub button_order: DialogButtonOrder,
       pub icon_theme: Option<Cow<'static, str>>,
   }

   impl ResolutionContext {
       /// Auto-detect from the current OS (calls detect::*).
       pub fn from_system() -> Self { /* ... */ }

       /// Deterministic values for tests (96 DPI, PrimaryRight, no icon_theme).
       pub fn for_tests() -> Self { /* ... */ }
   }
   ```
2. **No `impl Default`** per J.2's "signal intent" requirement.
3. Change `ThemeMode::into_resolved` signature from
   `(font_dpi: Option<f32>)` to `(ctx: &ResolutionContext)`.
4. Add `Theme::resolve_system() -> Result<ResolvedTheme>` shortcut
   that constructs `ResolutionContext::from_system()` internally.
5. `OverlaySource.font_dpi` becomes `context: ResolutionContext`.
6. Migrate `pipeline::from_system_inner` to build one `ResolutionContext`
   per theme-build and pass by reference.
7. Migrate all tests that currently pass `Some(96.0)` to
   `ResolutionContext::for_tests()`.

**Accessibility placement — per J.2 refinement on B4:** do **not**
add `accessibility: AccessibilityPreferences` to `ResolutionContext`.
It stays on `SystemTheme` (already implemented). Resolution-time
inputs vs render-time state are different concerns.

**Confidence:** high on design (doc-1-merge-reviewed). Medium on
migration scope — needs careful audit of every `into_resolved` /
`resolve` call site.

## G9 · Ownership-type uniformity for theme-name strings (P2)

**Source:** doc 2 §J.2 / §K.3 (lines 2603-2643):
> Pick one ownership type across the crate and use it consistently.
> `Arc<str>` with `serde`'s `rc` feature. Apply to:
> - `SystemTheme::name`
> - `ResolvedThemeVariant::icon_theme`
> - `ReaderOutput::name`
> - `ResolvedFontSpec::family`

**Status:** font family uses `Arc<str>` (`model/font.rs:235`). The other
three use `Cow<'static, str>`:

```
native-theme/src/lib.rs:347               SystemTheme::name:       Cow<'static, str>
native-theme/src/lib.rs:363               SystemTheme::icon_theme: Cow<'static, str>
native-theme/src/model/mod.rs:252         Theme::name:             Cow<'static, str>
native-theme/src/model/defaults.rs:125    ThemeDefaults::icon_theme: Option<Cow<'static, str>>
```

**The argument for the existing `Cow<'static, str>` choice:** bundled
preset names are `&'static str` literals. `Cow::Borrowed(static_lit)`
is zero allocation, zero refcount. `Arc<str>` requires at least one
allocation per unique string at construction time.

**The argument for J.2's `Arc<str>` recommendation:** cross-type
consistency. `ResolvedFontSpec::family` is `Arc<str>` precisely because
many widgets share the same font family string — dedup matters. The
same argument applies (weakly) to icon theme names: many resolved
widgets might carry the same icon-theme reference.

**Resolution direction:** under "perfect API" the consistency argument
wins if and only if the cost is small. Audit:

1. Count unique `name` strings across bundled presets: 16 presets × 2
   variants = 32 potential uses. In practice, **each resolved theme
   carries exactly one `name`** — no dedup benefit.
2. Count unique `icon_theme` strings: KDE has 2 (`breeze`,
   `breeze-dark`), others have 1. Per-resolved-theme: one icon_theme.
   No dedup benefit.
3. Count `family` strings: every resolved widget can carry a font
   family override. 26 widgets × connector count: real dedup benefit.
   **Keep `Arc<str>` for font family.**

**Recommended fix:** **accept the current `Cow<'static, str>` choice
as a principled deviation from J.2**, and document the decision:

1. Add a doc comment to `SystemTheme::name` at `lib.rs:347` noting
   the ownership-type trade-off explicitly.
2. Update doc 2 §J.2 / §K.3 with a post-script that explains why
   the uniform-`Arc<str>` recommendation was not adopted for the
   `name`/`icon_theme` axis (no dedup benefit, bundled presets are
   `'static`).
3. **No code change** to the existing types.

**Confidence:** high. This is actually a case where the implementation
is better than the doc's recommendation; the gap is purely
documentation-sync.

## G11 · Principled deviation — `cargo test --workspace` vs `./pre-release-check.sh` (P0 doc-only)

**Context:** Phase 93 verification (2026-04-19) flagged that `cargo test --workspace --all-features` fails to compile, blocking the plan-level must_have that claimed that command as a gate. Root-cause analysis follows.

**The error:**

```
error[E0277]: the trait bound `std::string::String: WriteColor` is not satisfied
   --> ~/.cargo/registry/src/index.crates.io-.../naga-27.0.3/src/front/wgsl/error.rs:113:20
```

Same error recurs at `naga-27.0.3/src/span.rs:326`. Three errors total, all in `naga` crate, all the same trait-bound mismatch against `WriteColor`.

**Root cause:**

- `naga 27.0.3` (transitively pulled by `gpui-component v0.5.1` via `wgpu 27.0.0`) uses `codespan_reporting::term::emit` with `&mut String` as the writer argument.
- `codespan-reporting 0.12.0` removed the `impl WriteColor for String` that `0.11.x` provided. `naga 27.0.3` was written against the 0.11.x behaviour and not yet migrated.
- The workspace's `Cargo.lock` pins `codespan-reporting = 0.12.0` (line 1064-1067), which is correct for every other consumer and cannot be forced back to 0.11.x without breaking them.

**Options considered:**

| Option | Description | Status | Why |
|--------|-------------|--------|-----|
| A | `cargo update -p naga --precise 27.0.4` | Not viable | naga 27.0.4 does not exist on crates.io (`cargo info naga@27.0.4` returns "could not find"). The next naga release is `28.x`, which `gpui-component v0.5.1`'s `27.x` pin rejects via semver. |
| B | Narrow the plan's must_have from `--workspace` to `-p native-theme` scope | Weak | This is technically what we end up with, but framed as "scope narrowing" it reads as hiding the problem rather than honestly documenting why the wider scope is not currently achievable. |
| C | Exclude `connectors/native-theme-gpui` from `[workspace] members` so `--workspace` bypasses it | Worse | Breaks developer ergonomics (workspace-wide `cargo check` misses the connector), propagates the upstream defect into our project layout, and is Option B with more collateral damage. |
| D | Document as a principled deviation and align the plan's acceptance criterion with the actual release gate (`./pre-release-check.sh`) | **Selected** | See below. |

**Option D rationale:**

The project's actual release gate is `./pre-release-check.sh`. That script runs per-crate tests (lines 287-294), NOT `cargo test --workspace`:

```bash
for crate in $WORKSPACE_CRATES; do
    if [ "$crate" = "native-theme-gpui" ]; then
        run_check_soft "Running tests ($crate)" cargo test -p "$crate"
    else
        run_check "Running tests ($crate)" cargo test -p "$crate"
    fi
done
```

The `run_check_soft` wrapper (lines 50-62) treats a failure in `native-theme-gpui` as a warning, not a blocker — this is the EXACT treatment the current upstream state requires, and it is how the release script has been shaped since the first gpui-connector ship (Phase 14-03, 2026-03-09).

The `--workspace` claim in Plan 93 must_haves was an aspirational generalisation that predated the discovery of the naga issue. The honest criterion, matching the actual release gate, is:

> `cargo test -p native-theme --all-features` + `cargo test -p native-theme-derive` + `cargo test -p native-theme-build` + `cargo test -p native-theme-iced` all pass, AND `cargo test -p native-theme-gpui` either passes OR is known-failing for a documented upstream reason.

**Re-evaluation trigger:**

When gpui-component ships a release that either (a) bumps naga past 27.0.3 to a codespan-reporting 0.12.0-compatible version, or (b) pins `codespan-reporting = 0.11.x` in its own Cargo.toml, run:

```bash
cargo update -p gpui-component
cargo test --workspace --all-features
```

If both succeed, this deviation is obsolete and the `--workspace` criterion returns. Track the upstream status at:

- https://github.com/gfx-rs/wgpu/issues (naga is part of wgpu)
- https://github.com/zed-industries/zed (gpui-component's upstream)

**No code change.** This is a documentation-of-intent update only, anchoring the plan's acceptance criterion to the script that is the actual CI truth.

**Priority:** P0 for v0.5.7 release (the `--workspace` claim in Phase 93 must_haves otherwise blocks verification). Documentation-only work, zero implementation risk.

**Confidence:** high. The root-cause analysis is evidence-backed (exact error, exact codespan-reporting behaviour change, exact cargo-info output proving Option A non-viability).

**Cross-reference:** `.planning/phases/93-docs-todo-v0-5-7-gaps-md/deferred-items.md` "naga v27.0.3 workspace build error" entry (logged during Plan 01 execution 2026-04-19) is the source signal for this documented deviation.
